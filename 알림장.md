# 알림장 (Alrimjang)
> 조직 공지 · 실시간 채팅 · 할 일 · 설문 통합 플랫폼 | 개인 프로젝트 | 2개월

---

## 프로젝트 개요

종이 알림장과 단체 채팅방이 혼재하는 환경에서 **공지 열람 여부 확인이 불가능하다는 구조적 문제**를 해결하기 위해 기획했습니다.
교사는 공지를 작성하고 대상을 지정해 배포하며, 학부모는 수신한 공지를 열람하면 읽음 처리됩니다.
서비스 전체 흐름(인증 → 공지 배포 → 읽음 추적 → 실시간 채팅 → 할 일 관리 → 설문 조사)을 **혼자 설계하고 구현**했습니다.

## 문제 해결 과정

### 문제 1. 공지 배포와 열람을 분리하는 2단계 추적 구조 설계

**문제 정의**
단순히 공지를 저장하는 것만으로는 "누구에게 전달됐는지"와 "누가 읽었는지"를 구분할 수 없었습니다. 배포했지만 아직 안 읽은 상태를 표현하려면, 배포 여부와 열람 여부가 별도로 관리되어야 합니다.

**해결 과정**
`notice_receipts` 테이블로 배포(수신) 기록을, `read_at` 컬럼으로 열람 기록을 분리했습니다. 공지 배포 시 대상 조건(`notice_targets`: 전체/역할/그룹)을 기반으로 수신 대상자를 추출한 뒤, 수신자 전원에 대해 수신 레코드를 일괄 생성합니다. 사용자가 공지를 열람하면 해당 레코드의 `read_at`을 갱신합니다.

```
공지 작성 → 대상 설정(notice_targets) → 배포(notice_receipts 생성) → 열람(read_at 갱신) → 읽음/안읽음 집계
```

대상 추출과 수신 레코드 생성은 `NoticeAudienceService`에 단일 책임으로 위임하고, 컨트롤러는 배포 요청만 전달하도록 계층을 분리했습니다.

**결과**
수신자별 읽음 상태를 정확히 구분할 수 있게 됐습니다. 관리자가 미열람자를 식별하고 재알림을 보낼 수 있는 기반이 마련됐습니다. 이후 그룹 조건 추가 시에도 컨트롤러 수정 없이 서비스 내부만 변경하면 됩니다.

### 문제 2. 채팅방 접근 제어 — 방 ID만으로 진입 가능한 보안 취약점 해결

**문제 정의**
초기 구현에서는 채팅방 ID(`roomId`)만 알면 누구든 메시지를 조회하고 전송할 수 있는 구조였습니다. DM 방에 제3자가 접근하거나, 소속되지 않은 그룹 채팅에 진입할 수 있는 보안 취약점이 존재했습니다.

**해결 과정**
`ChatRoomAccessService`를 도입해 DM과 그룹방의 접근 규칙을 통합했습니다.

- **DM 방**(`dm__a__b`): 방 토큰에 요청자의 username이 포함되어야만 접근 허용
- **그룹 방**(`grp__{groupId}`): `group_members` 테이블에 사용자가 소속되어 있고, 그룹 `type='CHAT'`인 경우만 허용

메시지 전송(`send`), 조회(`getRecent`), 읽음 처리(`markAsRead`), 할 일 생성/완료/확인 등 **모든 채팅 관련 API에서** `assertAccessible()`을 호출하도록 강제했습니다.

**결과**
방 ID 조작으로 인한 비인가 접근이 차단됐습니다. 접근 검증 로직이 서비스 한 곳에 집중되어 누락 위험이 줄었습니다.

### 문제 3. 채팅 할 일 상태 전이 — 권한별 상태 흐름 강제

**문제 정의**
채팅 내 할 일 기능에서 "생성 → 완료 → 확인"이라는 상태 흐름이 있었지만, 누구나 아무 상태로든 변경할 수 있으면 할 일 추적의 신뢰성이 깨집니다.

**해결 과정**
`PENDING → DONE → CONFIRMED` 상태 전이를 서비스 레이어에서 강제했습니다.

```
PENDING ──(담당자만 완료 가능)──→ DONE ──(생성자만 확인 가능)──→ CONFIRMED
```

- **담당자**(`assigneeUsername`)만 `PENDING → DONE` 전이 가능 (`completeTask`)
- **생성자**(`createdByUsername`)만 `DONE → CONFIRMED` 전이 가능 (`confirmTask`)
- DB 업데이트 시 `WHERE status = 'PENDING'` / `WHERE status = 'DONE'` 조건을 걸어, 이미 전이된 상태에서의 중복 처리를 방지

**결과**
권한 없는 사용자의 상태 변경이 차단되고, 상태 전이 규칙이 코드에 명시적으로 표현되어 유지보수가 용이해졌습니다.

### 문제 4. 설문 응답의 무결성 검증 — 5가지 검증 레이어

**문제 정의**
설문 응답 제출 시 기간 외 응답, 필수 문항 누락, 단일 선택 문항에 복수 선택, 존재하지 않는 옵션 ID 전송, 동일 사용자의 중복 응답 등 다양한 무결성 위반이 발생할 수 있었습니다.

**해결 과정**
`submitResponse` 메서드에서 5단계 검증을 수행합니다.

| 순서 | 검증 항목 | 처리 방식 |
|------|-----------|-----------|
| 1 | 기간 검증 | `start_at` ~ `end_at` 범위 내인지 확인 |
| 2 | 중복 응답 | `survey_responses`의 UNIQUE 제약조건 + 사전 체크 |
| 3 | 필수 문항 | `is_required=true`인 문항의 답변 존재 확인 |
| 4 | 단일 선택 | `type='SINGLE'`인 문항의 선택 개수가 1개인지 검증 |
| 5 | 유효 옵션 | 전송된 `optionId`가 해당 문항의 실제 옵션 ID에 포함되는지 확인 |

객관식은 투표수/비율을 집계하고, 주관식은 최근 응답 20건을 조회하는 관리자용 결과 화면도 구현했습니다.

**결과**
클라이언트 조작으로 인한 비정상 응답이 서비스 레이어에서 차단됩니다. 검증 로직이 한 메서드에 집중되어 테스트와 유지보수가 용이합니다.

### 문제 5. WebSocket 채팅에서 안읽은 메시지 수 실시간 동기화

**문제 정의**
채팅 목록 화면에서 안읽은 메시지 수를 표시해야 하는데, 페이지 로드 시점과 메시지 수신 시점 사이의 카운트 불일치가 발생했습니다. 페이지를 새로고침하지 않으면 카운트가 갱신되지 않는 문제였습니다.

**해결 과정**
STOMP 메시지 수신 이벤트를 클라이언트에서 구독하고, 메시지 수신 시 REST API(`/api/chat/rooms/{roomId}/unread-count`)를 호출해 카운트를 갱신하는 방식을 채택했습니다. 안읽은 카운트 계산은 `chat_room_reads.last_read_at` 이후의 메시지 수를 SQL로 집계합니다. 채팅방 입장 시에는 `markAsRead`로 읽음 시각을 갱신해 카운트를 0으로 초기화합니다.

**결과**
채팅 목록의 안읽은 메시지 수가 실시간으로 갱신되고, 채팅방 입장 시 즉시 초기화됩니다.

## 아키텍처

```
[Browser]
   │
   ├── HTTP (MVC)  →  Controller → Service → MyBatis Mapper → PostgreSQL
   │
   └── WebSocket (STOMP)
           └── /app/chat/send  →  ChatController  →  ChatService  →  DB
               /topic/chat.{roomId}  ←  클라이언트 구독
```

**계층 분리 원칙**
- **Controller**: 요청/응답 변환만 담당, 비즈니스 로직 포함 금지
- **Service**: 하나의 서비스 클래스는 하나의 도메인 책임만 수행 (접근 제어, 상태 전이, 검증)
- **Mapper**: SQL은 XML에 분리, 인터페이스로만 호출

## 핵심 DB 설계

```
users ─── groups ─── group_members

notices (공지)
    └── notice_targets   (배포 대상 조건: 전체/역할/그룹)
    └── notice_receipts  (수신 레코드 + 열람 시각)

chat_messages (채팅 메시지)
    └── chat_room_reads  (읽음 시각 기록)
    └── chat_tasks       (할 일: PENDING → DONE → CONFIRMED)

surveys (설문)
    └── survey_questions  (문항)
        └── survey_options  (선택지)
    └── survey_responses  (응답 헤더, UNIQUE(survey_id, user_id))
        └── survey_answers  (개별 답변)
```

**설계 포인트**
- `notice_receipts`의 `delivered_at`과 `read_at` 분리: 배포 여부와 열람 여부는 서로 다른 개념
- `chat_tasks`에 `created_by` / `assignee` 분리: 권한별 상태 전이 규칙의 기반
- `survey_responses`에 UNIQUE 제약조건: DB 레벨에서 중복 응답 방지
- 핵심 컬럼에 인덱스 12개 구성: 채팅/설문/공지 조회 성능 최적화

## 트러블슈팅

### 1. WebSocket STOMP 구독 시 접근 제어 누락 — HTTP 계층과 분리된 보안 구멍

**증상**
REST API(`/api/chat/rooms/{roomId}/...`)에는 `ChatRoomAccessService.assertAccessible()`로 접근 제어를 걸었지만, WebSocket STOMP 구독(`/topic/chat.{roomId}`)에는 접근 검증이 없었습니다. 방 ID를 알고 있는 사용자가 JavaScript 콘솔에서 직접 STOMP 구독을 걸면, 권한 없는 채팅방의 실시간 메시지를 수신할 수 있는 상태였습니다.

**원인 분석**
Spring Security의 `SecurityFilterChain`은 HTTP 요청에만 적용됩니다. WebSocket STOMP 메시지는 별도의 `MessageChannel`을 통해 처리되므로, HTTP 레벨 인가 설정이 STOMP 구독에는 전혀 영향을 미치지 않았습니다. 즉, HTTP와 WebSocket이 서로 다른 보안 경계를 가진다는 점을 간과한 것이 근본 원인이었습니다.

**조치**
`ChannelInterceptor`를 구현한 `ChatStompAccessInterceptor`를 작성하고, `WebSocketConfig.configureClientInboundChannel()`에 등록했습니다. 인터셉터는 `StompCommand.SUBSCRIBE` 시점에 destination에서 roomId를 추출한 뒤, `ChatRoomAccessService.assertAccessible()`을 호출하여 비인가 구독을 차단합니다. 메시지 전송(`SEND`)은 이미 컨트롤러(`@MessageMapping`)에서 검증하고 있었으므로, 구독 시점만 인터셉터로 보완하는 구조를 택했습니다.

```
SUBSCRIBE /topic/chat.{roomId}
    → ChatStompAccessInterceptor.preSend()
        → chatRoomAccessService.assertAccessible(roomId, principal.getName())
            → DM: 방 토큰에 username 포함 여부 검사
            → Group: group_members 소속 + type='CHAT' 검사
```

### 2. 채팅방 목록 조회의 N+1 쿼리 → CTE 기반 단일 쿼리 최적화

**증상**
채팅방 목록 API(`/api/chat/rooms`)에서 방 개수가 늘어날수록 응답 시간이 선형으로 증가했습니다. 초기 구현은 "내가 속한 방 목록 조회 → 각 방마다 마지막 메시지 조회 → 각 방마다 안읽은 메시지 수 조회"로 N+1 패턴이 발생하고 있었습니다.

**원인 분석**
방 목록을 먼저 가져온 후 Java 코드에서 방별로 `findLastMessage(roomId)`, `countUnread(roomId, userId)`를 반복 호출하는 구조였습니다. 방이 20개면 최소 41번의 쿼리가 실행되는 상황이었습니다.

**조치**
PostgreSQL의 CTE(Common Table Expression)와 `DISTINCT ON`을 활용해 단일 쿼리로 통합했습니다.

```sql
WITH my_rooms AS (     -- 1. 내 방 목록 (DM + 그룹 + 할 일 기반)
    SELECT DISTINCT room_id ...
    UNION ...
),
last_message AS (      -- 2. 방별 최근 메시지 (DISTINCT ON)
    SELECT DISTINCT ON (m.room_id) ...
),
last_task AS (         -- 3. 방별 최근 할 일
    SELECT DISTINCT ON (t.room_id) ...
),
room_base AS (         -- 4. 메시지/할 일 병합
    SELECT ... FROM my_rooms LEFT JOIN last_message LEFT JOIN last_task
)
SELECT room_id, last_message, unread_count ...  -- 5. 최종 결과 + 안읽음 서브쿼리
FROM room_base ORDER BY last_message_at DESC
```

메시지와 할 일 중 더 최근 것을 `CASE WHEN`으로 판별해 채팅방 목록의 마지막 메시지를 결정하고, 안읽은 카운트도 서브쿼리로 한 번에 계산합니다.

**결과**
방 개수와 관계없이 **쿼리 1회**로 목록 전체를 조회할 수 있게 됐습니다. `chat_messages(room_id, created_at DESC)` 인덱스가 `DISTINCT ON` 절의 성능을 지원합니다.

### 3. Spring Security CSRF 토큰과 WebSocket 엔드포인트 충돌

**증상**
WebSocket 연결(`/ws-chat`) 및 채팅 REST API(`/api/chat/**`) 호출 시 403 Forbidden 에러가 발생했습니다. 브라우저 개발자 도구에서 확인하면 CSRF 토큰 검증 실패가 원인이었습니다.

**원인 분석**
Spring Security는 기본적으로 모든 POST/PUT/DELETE 요청에 CSRF 토큰을 요구합니다. WebSocket 핸드셰이크는 HTTP 업그레이드 요청으로 시작되는데, SockJS 폴백 전송(`/ws-chat/xhr_send` 등)이 POST로 전송되면서 CSRF 검증에 걸렸습니다. 채팅 REST API도 JavaScript(`fetch`)에서 CSRF 토큰을 헤더에 포함하지 않으면 동일하게 차단됐습니다.

**조치**
WebSocket과 채팅 API 경로에만 선택적으로 CSRF를 비활성화했습니다.
```java
.csrf(csrf -> csrf.ignoringRequestMatchers("/ws-chat/**", "/api/chat/**"))
```
CSRF를 끈 대신, WebSocket 구독 시에는 `ChatStompAccessInterceptor`로, REST API에서는 `ChatRoomAccessService`로 각각 인가를 보장하여 보안 공백을 메웠습니다. 세션 기반 인증(`Principal`)은 WebSocket에서도 유지되므로, 인가 누락 없이 CSRF 충돌을 해결했습니다.

## 느낀 점

- **MyBatis XML Mapper**를 사용하면서 SQL을 직접 제어하는 경험을 쌓았고, 조인이 복잡한 조회(수신자별 읽음 상태 집계, 설문 투표율 계산)에서 명시적 SQL이 의도를 더 명확히 전달한다는 점을 깨달았습니다.
- **WebSocket 상태 관리**에서 서버 재시작 시 세션이 끊기는 문제를 확인했고, 추후 Redis Pub/Sub를 도입해 다중 인스턴스 환경에서도 채팅이 동작하도록 개선할 계획입니다.
- Spring Security의 **세션 기반 인증 흐름**(UsernamePasswordAuthenticationFilter → SecurityContext → CustomUserDetailsService)을 직접 구현하며 인증 체계 전반을 이해할 수 있었습니다.
- Flyway를 도입하며 **스키마 변경을 코드로 추적**하는 습관을 갖게 됐고, DB 사전 점검 스크립트를 만들어 배포 전 검증 프로세스를 정립했습니다.
